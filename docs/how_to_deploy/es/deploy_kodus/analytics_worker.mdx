El analytics worker es un add-on **exclusivo de Enterprise**. Ejecuta el
cron de ingestión que alimenta los dashboards del Cockpit (métricas tipo
DORA, ciclo de vida de PR, classifier de PR basado en LLM).

<Warning>
El instalador por defecto **no** incluye este worker. Los despliegues
self-hosted de la versión community no lo necesitan y estas vars se
filtran del `.env.example` por defecto. Detente aquí a menos que tengas
una licencia self-hosted Enterprise y quieras los reportes del Cockpit.
</Warning>

## Qué hace

Un proceso Node separado ejecutando la **misma imagen del `worker`**
(`kodus-ai-worker`), seleccionado en el boot vía `WORKER_ROLE=analytics`.
Dos crons disparan desde este proceso y solo desde él:

- **Ingestión** (`ANALYTICS_INGESTION_CRON`, default `*/30 * * * *`) — lee
  pull requests y sesiones de review desde Mongo + el Postgres OLTP, y los
  proyecta al schema `analytics`.
- **Classifier** (`ANALYTICS_CLASSIFIER_CRON`, default `*/15 * * * *`) —
  llama un LLM para etiquetar cada PR con un tipo (feature/bugfix/refactor/etc).

Aislarlo del `worker` principal mantiene el event loop de code review
libre de queries de ingestión largas.

## Topología

El warehouse de analytics es un **schema** de Postgres, no una base de
datos separada. Dos layouts soportados:

- **Postgres compartido (recomendado para self-hosted)** — deja las vars
  `ANALYTICS_PG_DB_*` **sin definir** (comentadas, **no** en vacío). El config
  loader cae en el fallback de las vars `API_PG_DB_*` y crea un schema
  `analytics` en la misma instancia. Una sola DB para respaldar y operar.
- **Postgres dedicado** — define el bloque `ANALYTICS_PG_DB_*` apuntando
  a otra instancia. Úsalo cuando quieras queries analíticas totalmente
  aisladas del write path del OLTP.


## Habilitando en self-hosted Enterprise

### 1. Activa el profile `analytics`

El servicio `worker-analytics` ya viene en el `docker-compose.yml` del
installer, detrás de un [profile de Compose](https://docs.docker.com/compose/profiles/)
opt-in — así que queda apagado en installs community. Actívalo agregando a tu
`.env`:

```bash
COMPOSE_PROFILES=analytics
```

Usa la **misma imagen que `worker`** — solo `WORKER_ROLE=analytics` (definido en
el servicio, en `docker-compose.yml`) lo cambia a modo ingestión. Deja
`WORKER_ROLE=code-review` en el `.env` (worker principal); el contenedor de
analytics lo sobrescribe.

<Note>
¿Usas un compose hecho a mano (sin el installer de Kodus)? Agrega el servicio:

```yaml
worker-analytics:
    image: ghcr.io/kodustech/kodus-ai-worker:${IMAGE_TAG:-latest}
    platform: linux/amd64
    container_name: kodus-worker-analytics
    profiles: ["analytics"]
    environment:
        - WORKER_ROLE=analytics
    networks: [shared-network, kodus-backend-services]
    restart: unless-stopped
    env_file: [.env]
    depends_on: [db_kodus_postgres, db_kodus_mongodb, rabbitmq]
```
</Note>

### 2. (Opcional) ajusta el bloque de analytics en `.env`

Los defaults funcionan para el layout de Postgres compartido — solo necesitas
este bloque para cambiar los crons o apuntar a un Postgres dedicado.

**Postgres compartido (recomendado):** deja las vars de conexión
`ANALYTICS_PG_DB_*` **sin definir** (comentadas). El loader cae en el fallback
de `API_PG_DB_*` y crea el schema `analytics` en la misma instancia.

<Warning>
**No** pongas `ANALYTICS_PG_DB_HOST=` en vacío — déjalo comentado. Un string
vacío se trata como "definido en nada" y puede cortocircuitar el fallback al
Postgres principal.
</Warning>

```bash
ANALYTICS_PG_DB_SCHEMA=analytics        # default; cámbialo solo para renombrar
ANALYTICS_INGESTION_CRON=*/30 * * * *   # default
ANALYTICS_CLASSIFIER_CRON=*/15 * * * *  # default
# ¿Sin LLM key configurada? Desactiva el classifier — las métricas DORA /
# ciclo de vida funcionan sin él:
# ANALYTICS_CLASSIFIER_DISABLED=true
```

**Postgres dedicado:**

```bash
ANALYTICS_PG_DB_HOST=tu-host-de-analytics
ANALYTICS_PG_DB_PORT=5432
ANALYTICS_PG_DB_USERNAME=analytics
ANALYTICS_PG_DB_PASSWORD=...
ANALYTICS_PG_DB_DATABASE=kodus_analytics
ANALYTICS_PG_DB_SCHEMA=analytics
```

### 3. Boot — las migrations corren automáticamente

```bash
COMPOSE_PROFILES=analytics docker compose up -d
```

`worker-analytics` comparte el mismo `prod-entrypoint.sh` que `api`/`worker`.
Con `RUN_MIGRATIONS=true` (default del installer), las migrations del warehouse
corren en el primer boot, creando el schema `analytics` y sus tablas. La primera
corrida de ingestión importa el historial de PRs que Kodus ya tiene en Mongo;
las siguientes son incrementales (por `updatedAt`).

## Referencia

| Variable | Descripción | Default |
|---|---|---|
| `COMPOSE_PROFILES` | Ponlo en `analytics` para levantar el worker. | _sin def._ |
| `WORKER_ROLE` | Debe estar en `analytics` en este contenedor. | _requerido_ |
| `ANALYTICS_PG_DB_HOST` | Host del Postgres de analytics. Sin definir → reutiliza el Postgres principal. | _sin def._ |
| `ANALYTICS_PG_DB_PORT` | Puerto del Postgres de analytics. | `5432` |
| `ANALYTICS_PG_DB_USERNAME` | Usuario del Postgres de analytics. Sin definir → reutiliza `API_PG_DB_USERNAME`. | _sin def._ |
| `ANALYTICS_PG_DB_PASSWORD` | Contraseña del Postgres de analytics. Sin definir → reutiliza `API_PG_DB_PASSWORD`. | _sin def._ |
| `ANALYTICS_PG_DB_DATABASE` | Base del Postgres de analytics. Sin definir → reutiliza `API_PG_DB_DATABASE`. | _sin def._ |
| `ANALYTICS_PG_DB_SCHEMA` | Nombre del schema de las tablas del warehouse. | `analytics` |
| `ANALYTICS_PG_POOL_MAX` | Límite superior del pool del Postgres de analytics. | `5` |
| `ANALYTICS_INGESTION_CRON` | Cron schedule de la ingestión (UTC). | `*/30 * * * *` |
| `ANALYTICS_CLASSIFIER_CRON` | Cron schedule del classifier de tipo de PR vía LLM (UTC). | `*/15 * * * *` |

### Pausando la ingestión (avanzado)

Para detener la ingestión en runtime sin remover el contenedor, define
`ANALYTICS_INGESTION_DISABLED=true` y/o `ANALYTICS_CLASSIFIER_DISABLED=true`
y reinicia `worker-analytics`. El cron sigue agendado pero cada tick
hace short-circuit.

## Verificando que funciona

Después del boot, sigue los logs del analytics worker:

```bash
docker compose logs -f worker-analytics
```

Debes ver líneas como `analytics ingestion done in NNNms — {...}` cada
30 minutos y `analytics classifier done ...` cada 15 minutos. Si no,
confirma que `WORKER_ROLE=analytics` está seteado solo en este
contenedor (no en el `worker` principal — ese debe permanecer en
`code-review`).
