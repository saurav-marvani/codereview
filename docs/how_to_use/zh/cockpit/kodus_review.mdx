## 什么是 Kodus 审查标签页

**Kodus 审查**标签页回答一个问题：*您的团队是否在根据 Kodus 的建议采取行动？*

与生产力标签页（衡量部署频率、PR 周期时间等通用交付指标）不同，Kodus 审查关注的是 Kodus 本身——有多少建议被实施、哪些类别和规则被采纳或被忽略，以及团队在哪些地方提出了异议。

每个图表和表格都遵循页面顶部的全局**仓库**和**日期范围**筛选器。点击某个仓库行、类别条形或规则，即可深入查看其背后的建议。

## 概要卡片

| 卡片 | 含义 |
| --- | --- |
| **实施率** | 在该周期内，团队实施的已发送建议的百分比 |
| **已发送建议数** | 在该周期内，Kodus 发送的建议数量（以 PR 评论形式） |
| **负面投票率** | 反应中为 👎 的占比，以及与上一周期相比的趋势 |
| **已合并 PR 中被忽略的严重建议** | 在已合并的 PR 上未被实施的严重建议——可采取行动的风险清单 |

## 实施率

核心指标。当一条建议在 PR 关闭时的最终状态为 `implemented` 或 `partially_implemented` 时，即计为**已实施**。

```
Implementation rate = implemented suggestions ÷ sent suggestions
```

范围规则——所有实施率图表均适用：

- 只有**已发送**的建议才会被计入（即实际作为 PR 评论发布的建议）。Kodus 在评论前过滤掉的草稿建议永远不会被计入。
- 建议归属于 **PR 关闭的那一周**，因为只有在 PR 合并后，实施状态才最终确定。

### 逐周对比

周度图表展示趋势，并提供一个切换选项：

- **总体** — 单条实施率折线。
- **按严重程度** — 每个严重程度（critical / high / medium / low）一条折线，从而可以查看更高严重程度的建议是否更容易被实施。

### 按类别和按严重程度

- **按类别** — 每个建议类别的已发送数与已实施数。点击某个条形可打开按该类别筛选的建议浏览器。
- **按严重程度** — 每个严重程度级别的实施率。预期应呈递减梯度（critical 比 low 实施得更多）。如果看起来持平或倒挂，则说明严重程度并未引导团队行动。

<Warning>
  **严重程度图表上的 "All / Kodus only" 切换很重要。** Kody 规则携带的是*您*在规则上设置的严重程度，而不是 Kodus 的风险评估。将两者混合会扭曲校准读数——例如，一批采纳率很高的 medium 级 Kody 规则，可能会让 Kodus 看起来低估了 medium 级别。切换到 **Kodus only**，即可查看 Kodus 自身的严重程度校准。基于极少量建议构建的条形会被淡化并标记 `*`——来自寥寥几条建议的 0% 或 100% 并不是真实信号。
</Warning>

## 负面反馈

反馈来自 Kodus 建议评论上的 👍 / 👎 反应。

- **负面投票率**（概要卡片）— `👎 ÷ (👍 + 👎)`，以及与上一周期相比的趋势。越低越好。
- **按类别** — 团队最不认同的地方。带有大量 👎 的类别是重新调优或禁用的候选项。
- **趋势** — 负面投票的逐周变化。

## 仓库——健康状况

按仓库列出的表格：审查的 PR 数、已发送建议数、实施率、👍/👎，以及**最薄弱的类别**（在满足最低样本量的前提下，该仓库中实施率最低的类别）。它展示了 Kodus 在哪些地方被采纳，又在哪些地方被忽略。

点击某一行会将整个驾驶舱聚焦到该仓库（与在仓库筛选器中选择它的效果相同）。

## Kody 规则——健康状况

展示每条 Kody 规则在该周期内的表现：触发次数、实施率、👍/👎 以及状态。此处仅显示激活的规则——已删除或未激活的规则会被排除，因为您无法对它们采取行动。

状态按规则计算，并依照以下优先级顺序：

| 状态 | 含义 | 建议操作 |
| --- | --- | --- |
| **Stale**（陈旧） | 在该周期内没有任何触发 | 重新评估该规则是否仍有必要 |
| **Low data**（数据不足） | 有触发，但次数过少（少于 5 次）而无法判断 | 等待更多数据 |
| **Noisy**（噪音） | 团队主动给它投反对票（≥ 3 个 👎 且 👎 多于 👍） | 该规则校准有误——重写或限定其范围（例如排除测试） |
| **Ignored**（被忽略） | 触发很多次但几乎没有被实施（≤ 20%） | 质疑其相关性——团队是否真的在意它？ |
| **Healthy**（健康） | 其他所有情况 | 无需操作 |

**Noisy** 与 **Ignored** 看起来相似，但需要采取的行动不同。*Ignored* 是被动的——规则触发了但没人实施；你无法判断它是噪音还是单纯被忽视。*Noisy* 是主动的异议——团队明确给它投了反对票，所以你知道它就是噪音。这就是为什么一条同时符合两者的规则会显示为 **Noisy**：反对票是更强、更可执行的信号。这些阈值是合理的默认值，可能会随时间调整。

## 建议浏览器

每一次深入查看——无论是点击类别条形、规则行，还是"被忽略的严重建议"卡片——都会打开**建议浏览器**：一个可筛选、分页的列表，展示数字背后的实际建议。

筛选条件：仓库、类别、严重程度、实施状态、Kody 规则以及自由文本搜索。每一行均可展开，显示现有代码与建议代码的对比，以及指向 PR 评论的链接。
