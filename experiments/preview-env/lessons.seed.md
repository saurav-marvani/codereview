- pkill -f '<pattern>' inside a playbook command kills the executing shell itself when the pattern appears verbatim in the command text (bash -lc carries the whole script in its cmdline). - Negated expectations in tests need careful double-checking: a failed assertion may be a test bug, not a product bug.
- Regex patterns are deliberately protected from lowercasing, which is a subtle correctness point worth explicitly verifying.
- PR VALIDATION: a plausible PR description induces confirmation bias — test the FALSIFYING input, not the description's happy path. When a diff transforms one side of a comparison but excludes the other (lowercases input but not the regex pattern; trims/casts one operand), the bug is the resulting ASYMMETRY: use an input that exposes it (uppercase literal pattern vs lowercased input), never one that already satisfies both sides (a /pattern/i already carries the i flag and hides a missing-i bug). Always test the negated operator too — predicate bugs invert into false positives.

- From-zero reproducibility (auto-replay gate is essential): replaying the agent's Kodus playbook on a TRULY fresh VM exposed a cascade of defects invisible on the detection VM — (1) uncaptured prerequisite: playbook started with 'pnpm run setup' but never installed pnpm (fresh VM has node/npm/corepack, NOT pnpm -> exit 127); (2) spurious host step: 'pnpm run build' = 'nest build' fails (can't resolve src/main.ts) because Kodus is a monorepo and the real build happens INSIDE docker; (3) host jest fails on clean/branch-mismatched env. KEY: the docker 'services' phase (the actual app) DID boot healthy on the fresh VM, so the environment reproduces via docker — it's the agent's bolted-on host build/test steps that don't. Product answer: (a) an auto-replay gate that replays the playbook on a fresh VM during detection and bounces failures back forces the agent to make the playbook self-sufficient (capture prereqs, drop spurious steps); (b) for docker-based apps the trustworthy 'it works' signal is docker service health, not fragile host-level build/test the agent adds. The agent under-captures prereqs it ran during exploration ('it just worked' steps don't reach the playbook).
- Playbook harness: each playbook command runs as a SEPARATE SSH session, so a background process started with 'nohup <svc> &' in one command DIES before the next command runs (the health check then fails with 'Couldn't connect'). Long-running services in a playbook must be started with setsid (setsid bash -c '...' < /dev/null &) so they survive the session, OR the service-start and its checks must be in the SAME command. The from-zero 'preview verify' gate catches exactly this (it caught the kutt playbook's nohup server not surviving to the health check).
- Auto-fix loop (preview harden): the loop structure works (reset->replay->LLM-fix->re-verify->teardown) and even surfaces harness bugs the playbook can't (found nohup-vs-setsid in the runner). BUT a single-shot fixer that re-emits the WHOLE playbook thrashes: it breaks phases that were passing (e.g. broke 'build' while trying to fix 'test'), oscillates, and doesn't precisely target the root-cause command. To converge, the fixer must be PATCH-BASED: change exactly one command in one phase, never touch phases that already passed (regression guard), and target the root cause (connection-refused -> fix the service START, add a readiness poll — not rewrite everything). Also cap re-emission size (full-playbook rewrites hit max_tokens and return unparseable YAML).
